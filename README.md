# Flodok

The operations OS for Indonesian teams — a single home for SOPs, contracts,
employee data, and your team's public portal. Bahasa-first, WIB/WITA/WIT,
Faktur Pajak ready.

This is the source for both the marketing site (flodok.com) and the
authenticated dashboard. End-user help docs live in
[src/pages/help/data.tsx](src/pages/help/data.tsx).

## Tech stack

- **Frontend**: Vite + React 19 + React Router 7, TypeScript, Tailwind 4
- **Backend**: Supabase (Postgres + Auth + Storage + Edge Functions)
- **Hosting**: Cloudflare Pages (static) + a sidecar Worker at
  `flodok-router/` for serverside routing concerns (incl. PDF rendering
  via Cloudflare Browser Rendering)
- **Editor**: TipTap with a custom bilingual schema
  (Document → Section → BilingualBlock with paired EN/ID sides). The
  structured doc lives in `content_doc` JSONB on `sops` / `contracts` /
  `document_templates`; flat markdown columns are a derived projection
  for legacy consumers (portal, signature hash, AI pipeline).
- **AI**: OpenRouter for translation (`translate-text`, used by the
  selection BubbleMenu) and structured-doc generation
  (`generate-document`, used by the "AI Generate" toolbar button).
- **Markdown rendering** (read-only surfaces only): react-markdown +
  remark-gfm + rehype-raw — for the portal renderer and Pending review.

## Repo layout

```
src/
  pages/dashboard/       Authenticated app pages (Hiring, Employees, Contracts, …)
  pages/public/          Public surfaces (Landing, Pricing, Portal)
  pages/help/data.tsx    User-facing help content (sections + topics + FAQ)
  components/            Shared UI primitives + feature-specific components
  lib/                   Pure helpers — no React. Each file is one concern.
  contexts/              React contexts (Language, Billing, Breadcrumbs)
  hooks/                 Reusable hooks
  types/
    database.ts          Generated by `supabase gen types typescript` — do not
                         edit by hand for schema changes; regenerate after
                         each migration. Hand-edits get clobbered.
    aliases.ts           Manually-maintained Row-type aliases. Add new aliases
                         here so they survive type regeneration.

supabase/
  migrations/            Numbered SQL migrations. Apply with `supabase db push`.
  functions/             Edge Functions (Deno). Deploy with `supabase functions deploy <name>`.

flodok-router/           Cloudflare Worker (separate wrangler config).
                         When running wrangler against this worker, always
                         pass `--config flodok-router/wrangler.toml`.
```

## Local development

```bash
npm install
npm run dev                # vite dev server, http://localhost:5173
npm run build              # tsc -b && vite build (also runs in CI)
npm run lint
npm run preview            # build then serve via wrangler (closer to prod)
```

You'll need `.env.local` with at minimum:

```
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<anon key>
```

The Supabase project is linked via `supabase link --project-ref <ref>`.
Run `supabase projects list` to confirm the linked project.

## Database changes

Migrations are numbered SQL files in `supabase/migrations/`. Convention:
`NNN_short_description.sql`. The number must be the next free integer.

Workflow:

```bash
# 1. Create the migration file by hand (no scaffolding command in use here).
#    Always include a top-of-file comment block explaining *why* — what
#    decision is being encoded. Future readers (including future-you and
#    future-Claude) need that context far more than they need to read the
#    DDL.

# 2. Dry-run against the linked remote.
supabase db push --dry-run

# 3. Apply.
supabase db push --yes
```

After applying, regenerate the typed schema:

```bash
supabase gen types typescript --project-id <ref> > src/types/database.ts
```

If you can't / don't regenerate immediately, hand-add the new fields to
`database.ts` — but mark a TODO and regenerate before merging. The hand-add
must match the generated shape (alphabetized within Row/Insert/Update).

## Documents (SOPs & Contracts)

Both doc types share one structured-document model and one editor.

- **Storage**: `content_doc` JSONB column on `sops`, `contracts`,
  `document_templates`, `sop_versions`, `contract_versions`. Source of
  truth from Phase C onward. Flat markdown columns (`content_markdown`,
  `content_markdown_id`) are a derived projection — the snapshot helper
  re-runs `docToMarkdown` on every save so portal/PDF/signature-hash
  consumers keep working.
- **Editor**:
  [src/components/editor/bilingual/DocumentEditor.tsx](src/components/editor/bilingual/DocumentEditor.tsx).
  TipTap with a custom schema (Document → Section → BilingualBlock →
  BlockBody) that pairs an EN and ID side per block. View toggle
  (stacked vs side-by-side) is per-user, per-doc, persisted in
  `document_view_prefs`.
- **Snapshot helper**: every save funnels through
  `supabase/functions/snapshot-sop` via [src/lib/snapshotApi.ts](src/lib/snapshotApi.ts).
  It writes the version row, derives markdown, and translates per-block
  diffs against the `translation_cache` table.
- **AI**: per-selection translation
  ([translate-text](supabase/functions/translate-text/index.ts)) and
  whole-doc generation
  ([generate-document](supabase/functions/generate-document/index.ts),
  JSON-mode bilingual output).
- **PDF**: the bilingual renderer is pre-rendered via
  `react-dom/server` and POSTed to the Cloudflare Worker
  ([flodok-router/src/pdf.ts](flodok-router/src/pdf.ts)) which prints
  it through Browser Rendering.
- **Templates**: see "Document templates" below.

## Edge Functions

Each function lives in `supabase/functions/<name>/index.ts`. Shared helpers
sit under `supabase/functions/_shared/` (auth, snapshotting, etc.).

Deploy a single function:

```bash
supabase functions deploy <name>
```

Auth patterns — pick one:

- **User JWT**: pass the user's `Authorization: Bearer <session.access_token>`
  header through. Use `createClient` with that header so RLS gates writes.
  Standard pattern for dashboard-originated calls.
- **Service role** (`getSupabaseAdmin()`): bypasses RLS. Only use after
  validating the caller's identity yourself (e.g. via slug+access_token from
  the public portal, or `validateApiKey()` for external API key callers).
  See `_shared/auth.ts` for helpers.

## Lifecycle & status (employees)

Employees have a `lifecycle_stage` column that's the source of truth for
where they sit in their journey:

```
prospective → shortlisted → offered → signed → active → separated
                                                  ↑
                                              talent_pool ←┘ (no-go branch)
```

The Recruitment page surfaces the funnel stages (`prospective` through `signed`
plus `talent_pool`); the Employees page filters to `active` and `separated`.

Two transitions happen automatically:

- **`prospective → shortlisted`** when the candidate completes their pre-offer
  *screening profile* (NIK, date of birth, gender, religion, marital status,
  address) in the portal — enforced server-side by the
  `portal_advance_to_shortlisted` RPC (migration 213). This is the "screening
  gate": the profile is a screening instrument, and finishing it is what
  surfaces the candidate for review.
- **`signed → active`** when the employee's `join_date` arrives — see
  `src/lib/lifecycleAdvance.ts`.

The candidate portal collects data in two waves — a light screening profile
pre-offer, then bank / tax / document details after signing (see
`src/components/portal/CandidateOnboarding.tsx`). Design notes live in
`docs/recruitment-pipeline-plan.md`.

The visible status **badge** is *derived* from `lifecycle_stage` plus dates
(probation, join, resign) — see `src/lib/employeeStatus.ts`. Do not write
new code that reads the legacy `employees.status` column; it stays for
backward compat with the original Employees status filter, but new code
should call `deriveEmployeeStatus()`.

## Document templates

Templates live in their own typed `document_templates` table with a
`type` discriminator (`'sop' | 'contract'`). They share the bilingual
structured doc shape (`content_doc`) with concrete documents but skip
versioning, signing, and employee linking — a template is a starter,
not an issued document. Contract templates also carry the structured
contract fields (wage, allowance, hours, days) so they pre-fill a fresh
offer.

A contract template can target a specific job position via
`template_for_position`. When "Make offer" runs on a candidate, Flodok
looks up a template for that position and instantiates it into a fresh
draft contract via `src/lib/contractTemplates.ts:buildContractFromTemplate`.

Editing happens at `/dashboard/document-templates/:id/edit`
([DocumentTemplateEdit.tsx](src/pages/dashboard/DocumentTemplateEdit.tsx))
— a slim editor without snapshots or signatures. New contract templates
can be seeded from a built-in PKWT or PKWTT bilingual starter
([src/lib/pkwtStarterDoc.ts](src/lib/pkwtStarterDoc.ts)).

Merge tags live in `src/lib/mergeFields.ts` — `{{employee_name}}`,
`{{base_wage_idr}}`, `{{contract_start_date}}`, etc. The same renderer
runs in the bilingual editor (per-side localisation), the PDF export,
and the candidate-side contract review in the portal onboarding flow.

## Signature evidence

`contract_signatures` records typed name + font + role + the document
context it was signed in: `consent_text` (verbatim wording the signer
agreed to), `document_hash` (SHA-256 of the contract content at sign
time, see `src/lib/signatureFingerprint.ts`), `user_agent`, `signer_email`,
`signer_phone`, and `ip_address` (populated server-side via the
`sign-contract-ip` edge function — the browser cannot read its own
public IP).

Three sign call sites all funnel through the same row shape:

- Dashboard employer sign — `src/pages/dashboard/ContractEdit.tsx`
- Existing-employee portal sign — `src/pages/public/Portal.tsx`
- Candidate onboarding sign — `src/components/portal/CandidateOnboarding.tsx`

## Translations

UI strings live in `src/lib/translations.ts` (one file, English + Bahasa
side-by-side). When adding a new screen, add both languages at the same
time — there's no fallback, a missing ID key surfaces as `undefined` in
the UI.

## End-user docs

Help content for the in-app `/help` route lives in
[src/pages/help/data.tsx](src/pages/help/data.tsx). Each topic is a
React node assembled from `<P>`, `<H3>`, `<Steps>`, `<Bullets>`, and
`<Callout>` building blocks defined at the top of the file. The
`sections` registry at the bottom owns the IA — order, grouping, slugs.

When you ship a user-facing feature, update `data.tsx` in the same PR.
Don't ship the feature first and let the docs lag.
